Injection

  • #1
  • Injection section!

When an attacker sends hostile data to an interpreter

  • Occurs when untrusted data is sent to an interpreter as part of a command or query
  • Tricks the interpreter into executing unintended commands or accessing data without proper authorization
  • Obviously, we've all heard of SQL injection

Types of Injection

SQL

LDAP

XPath

NoSQL queries

And many more...

  • Some of the more common injections are SQL, NoSQL, OS command, Object Relational Mapping (ORM), LDAP, and Expression Language (EL) or Object Graph Navigation Library (OGNL) injection.
  • SQL
  • NoSQL
  • OS Command
  • Object Relational Mapping (ORM)
  • Lightweight Directory Access Protocol (LDAP) - similar to SQL but against Active Directory
  • Expression Language (EL) - used in Java web applications for embedding expressions into web pages
  • Object Graph Navigation Library (OGNL) - more Java

What could I be doing wrong?

Blindly accepting input

Concatenation

Impacts

Data Loss

Data Corruption

Data Disclosure

Loss of Accountability

Denial of Access

Prevention

Validate input on the backend

  • There are a couple ways you could do this
  • Validate directly, checking to see if the input matches a set string
  • Use explicit types

							[HttpGet]
							public void Get([FromQuery] string search)
							{
								var query = "SELECT * FROM Pets WHERE Id = " + search;

								// Execute the query
							}
  • So here's a pretty good example
  • Obviously, there are a few things that could be done differently here
  • We have an API endpoint that is searching for a pet by ID
  • But not very well
  • I'm assuming that most of us wouldn't do it like this

search = "1"


							SELECT * FROM Pets WHERE Id = 1

search = "1 OR 1=1"


							SELECT * FROM Pets WHERE Id = 1 OR 1=1
  • So what happens if it's called like it's supposed to be?
  • Everything's great!
  • BUT, what if someone calls the endpoint maliciously?
  • We have a different story
  • This is what our concatenated SQL query would look like
  • So what does this mean?
  • 1=1 pretty much invalidates our query and everything is returned

							[HttpGet]
							public void Get([FromQuery] int search)
							{
								var query = "SELECT * FROM Pets WHERE Id = " + search;

								// Execute the query
							}
  • Let's add a data type
  • Now, our attacker literally CAN'T pass in the malicious string
  • It HAS to be an ID

Use Entity Framework

  • Another great option is to use an ORM
  • That way, everything is done for us

							[HttpGet]
							public void Get([FromQuery] int id)
							{
								var pet = _context
									.Pets
									.Find(id);

								// Do other stuff
							}
  • So that takes our SQL query, and allows us to write it like this
  • Our threat disappears

							[HttpGet]
							public void Get([FromQuery] string search)
							{
								var pets = _context
									.Pets
									.Where(pet => pet
										.Name == search);

								// Do other stuff
							}
  • Even if we change the data type back to a string, EF still thwarts the attack
  • Linq queries are not created using concatenation, so we don't need to worry about injection here

Broken Authentication

  • #2
  • This is what we want to say, but it doesn't always happen

Exactly what it sounds like...

  • Your auth is broken...?

What could I be doing wrong?

Allowing credential stuffing

Allowing brute force

Bad password restrictions & recovery

  • Credential stuffing is the automated injection of breached username/password pairs in order to fraudulently gain access to user accounts.
  • Basically shoving breached usernames and passwords until something works
  • Brute force - basically trying again and again until something works, maybe with a list of passwords?

Impacts

Depends on your app...

Leak sensitive information

Money laundering

Identity theft

  • Could be this or this or that, just depends on what your software does

Prevention

Remove default logins

  • Convenient for testing, but also for attackers

admin

P@ssw0rd

  • We've all done it
  • Do not publish with this as an active user account!

Password restrictions

  • Minimum length
  • Maximum age
  • History
  • Complexity

ASP.NET Identity

  • Can do a lot of what you need for you
  • Easy to get up and running

							services.Configure<IdentityOptions>(options =>
								{
									// Default Password settings.
									options.Password.RequireDigit = true;
									options.Password.RequireLowercase = true;
									options.Password.RequireNonAlphanumeric = true;
									options.Password.RequireUppercase = true;
									options.Password.RequiredLength = 6;
									options.Password.RequiredUniqueChars = 1;
								});
  • Here's how you might set it up in a .NET Core 2.0 app
  • This would go in the Startup.cs

Multi-factor authentication!

  • Use it!!!!
  • Provides extra security
  • Can easily be set up with ASP.NET Identity

Sensitive Data Exposure

  • #3
Not encrypting the right stuff

What could I be doing wrong?

Sending data in plain text

Weak key generation

Weak storage techniques

Impacts

Comprised data

Personal information

Health records

Credit cards

Prevention

Store only necessary data

Use HTTPS

  • Using HTTPS ensures that all data being sent between the front end and api is encrypted
  • There is a setting in the Azure Portal under the Configuration section
  • Super easy to set
  • Redirects from HTTP to HTTPS

Disable caching

  • Another great way lock down your data would be to prevent caching
  • No data will be left stored on the front end

							[ResponseCache(Duration = 0, 
								Location = ResponseCacheLocation.None, NoStore = true)]
							public class PetsController : Controller
							{
								// control the things
							}
  • You can disable cahcing on an endpoint
  • Simply by decorating with this attribute
  • .NET is awesome!

XML External Entities (XXE)

  • #4

Exploitation of vulnerable XML processors

What could I be doing wrong?

Accepting XML directly

Not testing

Impacts

Data can be extracted

Execution of remote requests

Scan internal systems

DOS attacks

Prevention

  • Definitely the best way to prevent XXE attacks...

Don't use XML

  • Whatever you're using XML for, chances are there's a better way to do it

Don't allow external entities to resolve at all

  • The real problem comes in when you allow external entities to resolve
  • Luckily enough, you can turn that off

							var xmlDocument = new XmlDocument();
							xmlDocument.XmlResolver = null;
  • So this is just a new instance of System.Xml.XmlDocument
  • And configuring it to NOT load external entities is as simple as setting this property to null!

Keep libraries updated

  • Not much to be said here, just keep any third party XML processors up to date

Broken Access Control

  • #5

Users being able to perform unauthorized actions

  • It's a problem if users are able to get around your access control....

What could I be doing wrong?

Not testing

  • There are a multitude of ways to test access control, both from the perspective of the front and back end
  • You can even automate some of it

Impacts

Attackers can act as other users

Ability to create, read, update, and delete records

  • "Other users" may include admins
  • Could do anything that user could
  • CRUD access means they could decide to wipe your data

Prevention

Deny by default

  • Definitely don't set your application up to allow everything unless otherwise specified
  • It may be easier for testing, but it's a recipe for disaster

Record ownership

  • Great because it limits who can modify records

							[Authorize(Roles = "Administrator")]
							public class PetsController : Controller
							{
								// control the things
							}
  • So here, only users with the admin role can call this endpoint
  • Meaning no other role would be able to modify records within

Log access control failures

  • Multiple ways to log
  • Will mention some of the more common libraries later
  • Helps you to locate and eliminate threats

ASP.NET Identity

Third party loggers

Cross-Site Scripting (XSS)

  • #7
Injecting malicious scripts in order for data to be sent elsewhere

Types of XSS

Reflected

Stored

DOM

  • Reflected - The app includes unvalidated and unescaped user input as part of HTML output
  • Stored (critical) - The app stores unsanitized user input that is viewed at a later time by another user or an administrator
  • DOM - Sending attacker-controllable data to unsafe JS APIs

What could I be doing wrong?

Blindly accepting input again

Impacts

Session stealing

Account takeover

MFA bypass

And more...

  • Moderate for reflected and DOM
  • Severe for stored
  • More
  • DOM Node Replacement
  • Attacks against user's browser
  • Malicious software downloads
  • Key logging

Prevention

Escape untrusted input

  • If you're working with .NET and Razor, this gets easy

							@{
								var untrustedInput = "<\"123\">";
							}
						 
							@untrustedInput
  • You can see the untrusted input at the top
  • If we use that with an @, this is the output on the page
  • So you can see that it escapes the input

Enable CSP

  • Content Security Policy
  • Basically shuts out any other scrips that would be loaded from anywhere else

							app.Use(async (context, next) =>
							{
								context.Response.Headers.Add(
									"Content-Security-Policy",
									"script-src 'self';");

								await next();
							});
  • So here, we're just saying not to load any scripts but the ones we have

Insecure Deserialization

  • #8

Deserializing untrusted, potentially malicious objects

What could I be doing wrong?

Deserializing anything and everything

  • Accepting raw user input seems to be a theme

Impacts

Remote code execution

Ability to create, read, update, and delete records

Prevention

Don't serialize untrusted objects

  • You don't want to be deserializing random data supplied by the user

Type constraints


							Pet pet = JsonConvert.DeserializeObject<Pet>(jsonString);
  • This makes our lives better, as now we KNOW what we're deserializing to
  • The string has to be deserialized into the object we provide
  • Can't rely on this

Log deserialization exceptions

  • Really, we should be logging all exceptions
  • We will be in the know if anyone is trying to exploit any part of the app that deserializes data
